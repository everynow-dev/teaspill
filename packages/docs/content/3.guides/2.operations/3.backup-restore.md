teaspill keeps its state in three stores, and they hold different kinds of truth. Backing up is straightforward; the part worth understanding is what happens when you restore *some* of them and not others — because the answer is deliberate, and knowing it lets you pick the right trade-off under pressure.

## What each store owns

| Store | Holds | Durability role |
|---|---|---|
| Postgres | The [catalog](/concepts/projections-catalog): the registry row for every entity, plus the **archive of record** — the complete stored state of every agent that has ever [archived](/concepts/lifecycle). | Archival by design. This is the durable source of truth for archived agents. |
| durable-streams | The history store: every [timeline](/concepts/timelines-events) and its live delta and workspace-output streams. Append-only, never read to decide what an agent does next. | The readable record. Losing it costs history, not control. |
| Restate | The **live working state** of every *active* agent — its in-flight conversation, its `seq` counter, its pending work. This is control-flow state, not an archive. | Deliberately a working cache, not a durable archive. |

The asymmetry in that last column drives everything below: **Postgres is archival by design; Restate is deliberately not.** An archived agent's full state lives in Postgres, so losing Restate is survivable for it. An agent that was still active kept its working state only in Restate — so losing Restate loses that agent's in-flight state, and there is nowhere else to recover it from. That is the intended cost model, not a gap in it.

## Taking a backup

`scripts/backup.sh` captures all three stores from a running stack: a Postgres dump plus filesystem snapshots of the history and coordinator volumes.

```sh
# Full backup of the running stack into a timestamped directory:
scripts/backup.sh -d ./backups/2026-07-19
```

By default the backup is **quiesced**. Postgres is dumped live (its dump is always internally consistent), then the coordinator and history-store containers are briefly stopped while their volumes are copied, then restarted. The pause is typically seconds; wakes that arrive during it are retried by their callers, not lost. The result is a true point-in-time backup — all three stores reflect the same instant.

The `--live` flag skips the pause for zero downtime, at a cost:

::caution
`--live` copies the coordinator and history-store volumes while they are being written. The three stores' copies then land at slightly different instants — a **torn backup**. The catalog can end up ahead of or behind what the history copy actually holds, and the coordinator's own on-disk state may be caught mid-write. The catalog-versus-history skew is the kind the running stack repairs itself once restored; the coordinator's internal consistency is not something it can repair after the fact. Use quiesced mode for anything you would actually restore from.
::

Useful flags: `-d`/`--dir` sets the output directory (defaults to a timestamped directory under `backups/`), `-p`/`--project` and `-f`/`--file` override the Compose project name and file, and `--live` selects the torn-backup mode. Each backup directory gets a `MANIFEST.txt` the restore script reads.

## Restoring

`scripts/restore.sh` restores from a backup directory. It can restore all three stores or any subset — and the subset you pick is exactly what determines whether recovery is clean.

```sh
# Full, clean recovery:
scripts/restore.sh -d ./backups/2026-07-19 --all

# A subset (read the matrix below before choosing this):
scripts/restore.sh -d ./backups/2026-07-19 --postgres --streams
```

Select stores with `--postgres`, `--streams`, `--restate`, or `--all` (the default when you name none). `-y`/`--yes` skips the confirmation prompt for scripted use; `-p`/`--project` and `-f`/`--file` mirror the backup script.

::caution
Restore is destructive to the stores it targets: it drops and recreates the catalog's objects and wholesale-overwrites the history and coordinator volumes. There is no undo — take a fresh backup first if you might want to roll back the restore itself.
::

## What combinations restore cleanly

This is the load-bearing part. Restoring fewer than all three stores is supported, but each combination has a defined outcome.

| Restored | Outcome |
|---|---|
| **All three** | Clean full recovery. The stack comes back exactly as it was. |
| **Catalog + history, no coordinator** | Archived agents fully recover; agents that were still active are lost — loudly. |
| **Coordinator + catalog, no history** | Everything keeps running; the timeline shows a marked gap where history is missing. |
| **Coordinator only, or history only** | Don't. Restore the catalog alongside either. |

### Catalog + history, without the coordinator

Every agent that had **archived** at or before backup time recovers completely. Its full state lives in the catalog as the archive of record, so a message to it after restore [resurrects](/concepts/lifecycle) it — same identity, same timeline, transparent to the caller.

Every agent that was **active or idle but never archived** is lost. Its working state existed only in the coordinator, which you did not restore. Crucially, this failure is *loud*, not silent: a message to such an agent finds no live state and no archived state to recover from, and the send **fails visibly** — the sender sees an error rather than a message quietly vanishing. The agent's catalog row still lists it and its history is still readable, but it cannot be woken again.

If that class of loss is unacceptable for your deployment, you have two levers: always restore the coordinator alongside the catalog and history, or shorten the idle-archive window (set [`TEASPILL_IDLE_ARCHIVE_MS`](/reference/configuration#agent-loop-and-executor) below the 30-minute default) so that less state is ever only-in-the-coordinator at any moment — the more aggressively agents archive, the more of them are safe in the catalog.

::note
The coordinator also holds your **deployment registrations** — which agent-loop service serves which agent type. Any restore that leaves the coordinator empty (this combination, or a fresh coordinator volume) loses them too, so no spawn or send can route until you **re-register your deployments** — re-run `teaspill dev` or whatever registration step your deployment uses. A full restore brings the registrations back with the rest of the coordinator's state, so this only applies when you deliberately skip or reset it.
::

### Coordinator + catalog, without history

The inverse. Live working state and the registry come back, but the history store does not. **Nothing about control flow breaks** — history is never read to decide what an agent does, so agents keep running exactly as before. What is lost is the readable record for the affected window: a UI joining or replaying an affected timeline hits a gap.

teaspill handles this honestly rather than pretending. When the stack detects that history is missing where the catalog says it should exist, it writes a recovery snapshot that marks a **history hole** — a deliberate, labeled gap that says "history is missing here" instead of silently presenting an incomplete timeline. Clients treat it as a sanctioned jump and resume from there; your UI can render "history gap" at that point. Control flow survives unmodified; the timeline carries a documented hole.

### Any other subset

Restoring the coordinator or the history store *without* the catalog leaves you with no registry: nothing can be listed or woken. Restore the catalog alongside either for anything operational. The restore script does not block these combinations — it prints a warning and proceeds — but treat them as recovery of raw data, not a working deployment.

## Scheduling backups

teaspill does not ship a scheduler — `scripts/backup.sh` is a primitive you invoke. Wiring it to cron (or your platform's scheduled-job mechanism) is a deployment decision, because backup frequency and retention depend on how much in-flight work you are willing to lose and where you store the output. A quiesced backup on a cron schedule, shipped off-host, is a reasonable default.

::tip
A backup you have never restored is a hypothesis, not a backup. Periodically restore into a throwaway stack and confirm the agents you care about come back — especially if you rely on the partial-restore behaviors above.
::
