## (b) pi provider soak → BUFFERED_PROVIDERS

Direct provider probes (streaming + buffered + single tool call), plus the full
`createPiHarness` loop with the real platform tools.

| provider / model | stream text | buffered text | stream tool call | full harness loop |
|---|---|---|---|---|
| opencode-go / deepseek-v4-flash | ok (2 text deltas) | ok | ok (14 tool_input deltas, args `{a:17,b:25}`) | ok — spawn→wait, success |
| google / gemini-flash-lite-latest | ok (2 text deltas) | ok | ok (1 tool_input delta, correct args) | ERROR (see below) |

**Verdict (operator design change): NO static allowlist.** The `BUFFERED_PROVIDERS`
set was removed as a drift risk and replaced by a graceful per-call fallback in
`createPiAiStepClient.step()`: stream by default; on a NON-abort stream throw, transparently
re-run the same turn buffered (identical journaled final) and set a sticky per-client flag so
later steps skip the wasted stream attempt; aborts never fall back; an explicit
`opts.buffered:true` remains the only force-buffer knob (deployment config, for a provider whose
streamed final silently diverges without throwing). This soak — both providers stream cleanly;
google's failures reproduce under buffered — is the evidence the change is a pure improvement.
Neither provider's *streaming* is unreliable.

Google fails the full harness loop for reasons that are NOT streaming problems and that
buffering (`completeSimple`, identical schema/messages) would not fix:

1. **Recursive tool schema** — `spawn_agent.args` / `finish.result` use the recursive
   `jsonValueSchema` (`$ref`/`$defs`). Gemini rejects it at declaration:
   `500 INTERNAL "Limits exceeded while trying to flatten schema. Schema is too complex to process."`
   (any toolset containing spawn_agent or finish breaks Google.)
2. **Missing thought_signature** — a multi-turn tool loop (send_message → tool_result →
   next turn) is rejected:
   `400 "Function call is missing a thought_signature in functionCall parts."`
   Gemini requires echoing a thought_signature on assistant `functionCall` parts;
   `toPiAiMessages` does not thread it.

Both are provider-integration gaps (schema shape + protocol), orthogonal to the
streaming/buffered choice. See follow-up candidate below.

## (c) tool-description tuning

Traps run through the real harness loop over opencode-go (the working small model):

- **implicit-report trap** ("Spawn a researcher … then report the findings"): spawn → wait.
  No fabrication. Correct.
- **fabricate trap** ("Spawn a calculator to compute 47*89. Then tell me the answer"):
  spawn → wait; said "Let me wait for its result." Did NOT fabricate. Correct.
- **poll trap** ("Spawn … then check whether it has finished and report its result"):
  spawn → list_children → wait. The model narrated the async model correctly
  ("the result doesn't arrive immediately — it will come later as a child_finished
  message"), called list_children ONCE (obeying the explicit "check" instruction),
  saw the empty catalog, and fell through to `wait`. No repeated polling, no fabrication.

**Signal**: the descriptions already teach the async spawn/wake model effectively to a
small reasoning model — no harmful misuse (fabrication, busy-loop, never-yield,
result-from-spawn-return) was observed. The one friction was reaching for
`list_children` as a "check for results" step under an explicit "check" prompt.

**Change**: strengthened `PLATFORM_TOOL_DESCRIPTIONS.list_children` with explicit
anti-poll teaching ("NOT a way to poll or check whether a child has finished … end your
turn (`wait`)"). Re-running the poll trap 2× after the edit: the model still calls
list_children once (it is obeying the direct "check" instruction — prompt compliance, not
a description bug) but continues to end correctly with `wait` and never fabricates. The
edit is a correct defensive reinforcement; teaching-substring test added to pin it.

## Follow-up candidate (not scheduled)

**Native-harness Google (Gemini) provider support** — to make `google` usable for
tool-calling agents: (1) a non-recursive tool-args schema (or a Gemini schema
transform that flattens `jsonValueSchema`), and (2) threading Gemini's
`thought_signature` through `PiAssistantBlock`/`toPiAiMessages` for multi-turn tool
loops. Sizeable; its own task. Until then, Google works for text-only / single-tool
agents but not multi-step tool loops through the native harness.
